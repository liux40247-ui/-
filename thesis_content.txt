[22] [Heading 1] 	摘  要	

[23] [Normal] 随着人工智能技术在图像识别、文本分类、智能推荐和工业质检等场景中的持续落地，分类模型的性能评估已经不再局限于单一准确率指标，而是逐渐转向准确性、鲁棒性、推理效率、资源消耗和结果可解释性的综合衡量。针对传统评估流程人工参与多、实验条件难以复现、评估结果缺乏统一对比口径以及可视化表达不足等问题，本文设计并实现了一套面向智能分类模型的性能评估系统。系统采用前后端分离架构，前端基于 Vue3与 ECharts 构建交互界面和可视化看板，后端基于 Python Web 服务、任务调度、模型适配器和 MySQL 数据库完成模型管理、数据集管理、评估任务执行、指标计算与报告生成。

[24] [Normal] 本文围绕系统需求分析、总体架构设计、核心模块实现和测试验证展开研究。系统构建了包括 Accuracy、Precision、Recall、F1-score、混淆矩阵、推理时延、吞吐量和资源占用在内的多维指标体系，并通过适配器模式提升 PyTorch、ONNX 等模型接入的扩展能力。实验结果表明，系统能够完成模型上传、数据集校验、参数配置、自动评估、图表展示和报告导出等主要流程，在典型分类任务中具备良好的可用性、稳定性与扩展性，可为智能分类模型的选型、优化和部署提供客观依据。

[25] [Normal] 关键词：智能分类模型；性能评估；指标体系；可视化分析；系统设计

[27] [Heading 1] ABSTRACT

[28] [Normal] With the rapid application of artificial intelligence in image recognition, text classification, intelligent recommendation and industrial inspection, the evaluation of classification models has moved from a single accuracy-oriented method to a comprehensive assessment involving accuracy, robustness, inference efficiency, resource consumption and interpretability. To solve the problems of heavy manual intervention, poor reproducibility, inconsistent comparison criteria and insufficient visualization in traditional evaluation workflows, this thesis designs and implements a performance evaluation system for intelligent classification models.

[29] [Normal] The system adopts a front-end and back-end separated architecture. The front end is implemented with Vue3 and ECharts, while the back end provides model management, dataset management, evaluation scheduling, metric calculation and report generation based on Python Web services, task queues, model adapters and MySQL. A multidimensional metric system is built, including Accuracy, Precision, Recall, F1-score, confusion matrix, inference latency, throughput and resource usage. Test results show that the system can support model upload, dataset validation, parameter configuration, automatic evaluation, visual analysis and report export, providing an objective basis for model selection and optimization.

[30] [Normal] Key words: intelligent classification model; performance evaluation; metric system; visualization; system design

[65] [Heading 2] 1.1 研究背景与意义

[66] [Normal] 智能分类模型是当前人工智能应用中最基础也最常见的模型类型之一。从图像识别中的目标类别判定，到自然语言处理中的情感分类、垃圾邮件过滤，再到工业场景中的缺陷识别和医疗场景中的辅助诊断，分类模型直接影响业务系统的决策质量。随着深度学习模型规模不断扩大、训练数据来源不断多样化，模型性能已经不能只依赖单次实验的准确率进行判断。模型在不同数据分布、不同运行设备、不同输入噪声以及不同类别比例下的表现差异，会直接影响模型能否稳定部署到真实业务环境中。

[67] [Normal] 开题报告中指出，现有评估工作普遍存在三个不足：一是评估指标偏单一，多数工具重点展示 Accuracy、Precision 等基础指标，对鲁棒性、泛化能力和资源消耗关注不足；二是评估流程缺乏标准化，不同研究团队所采用的数据集、实验环境和参数配置不一致，导致模型结果难以横向比较；三是评估工具自动化程度不高，模型导入、数据处理、指标计算和报告整理往往需要人工串联，效率低且容易引入人为误差。因此，围绕智能分类模型构建统一、自动化、可扩展的性能评估系统，具有明确的工程价值和应用意义。

[68] [Normal] 本文研究的意义主要体现在两个方面。从理论层面看，系统将准确性指标、稳定性指标、效率指标和资源指标组织为统一的评价体系，使分类模型的评价从“单点结果”扩展为“多维画像”。这种方式能够更全面地反映模型在真实部署环境中的可用性，为模型选型和迭代优化提供依据。从工程层面看，系统通过前后端分离、任务调度、模型适配器、可视化图表和报告导出等机制，将原本分散的评估步骤封装为可复用流程，能够降低研发人员和测试人员的使用门槛，提高评估工作的规范性和复现性。

[69] [Heading 2] 1.2 国内外研究现状

[70] [Normal] 在模型评估指标方面，传统机器学习研究已经形成了准确率、精确率、召回率、F1 值、ROC 曲线、AUC 值等较成熟的指标体系。对于类别不平衡问题，研究者通常采用宏平均、微平均、加权平均、混淆矩阵和类别级召回率等方式分析模型对少数类样本的识别能力。深度学习模型广泛应用后，模型评估又进一步关注推理时延、吞吐量、模型参数量、显存占用和部署平台差异等工程指标。对于需要上线运行的分类模型而言，推理速度和资源占用常常与准确率同等重要。

[71] [Normal] 在评估工具方面，Scikit-learn、PyTorch Lightning、TensorBoard、Weights & Biases 等工具提供了较丰富的实验记录和指标展示能力，但这些工具往往面向模型训练过程或实验管理过程，直接服务于“多模型、多数据集、统一流程评估”的能力仍需要二次开发。部分企业级机器学习平台具备模型注册、部署和监控能力，但通常系统规模较大、部署复杂，难以作为轻量化毕业设计系统直接复用。国内相关研究则更多聚焦于特定领域模型评估，例如文本分类、图像识别、医学影像诊断等，通用化评估流程和可扩展架构仍有进一步完善空间。

[72] [Normal] 综合来看，现有研究和工具为本文提供了指标体系、可视化方法和系统架构方面的参考，但仍存在评估流程分散、框架适配不统一、结果解释不直观、报告生成不便等问题。本文面向毕业设计的规模和实际可实现性，选择构建一套轻量化 Web 评估系统，重点解决模型接入、数据集管理、评估任务调度、核心指标计算、结果可视化和报告导出六类问题。

[79] [Heading 1] 第2章 系统需求分析与总体设计

[80] [Heading 2] 2.1 需求分析

[81] [Normal] 系统用户主要包括模型研发人员、课程或实验平台管理人员以及需要进行模型选型的普通使用者。模型研发人员关注不同模型在同一数据集上的效果差异，需要快速上传模型、配置数据集、选择指标并查看可视化结果；平台管理人员关注任务运行状态、资源占用、历史评估记录和报告归档；普通使用者则希望系统降低参数配置复杂度，以向导式流程完成评估并获得清晰结论。

[82] [Normal] 根据开题报告中的研究目标和中期汇报中的实现进展，系统功能需求可归纳为五大模块。模型管理模块负责模型文件上传、框架类型识别、版本记录和模型元数据维护；数据集管理模块负责数据集上传、样本校验、标签映射和基础预处理；评估核心模块负责指标计算、任务调度、并行处理和异常处理；结果可视化模块负责混淆矩阵、柱状图、折线图、雷达图和性能报告展示；系统管理模块负责用户权限、日志记录、参数模板和报告导出。

[83] [Normal] 非功能需求方面，系统需要具备可用性、稳定性、扩展性和可维护性。可用性要求界面流程清晰，上传、配置、执行和查看结果的路径尽量简短；稳定性要求任务异常时能够记录错误并给出可理解提示；扩展性要求后续能够接入 TensorFlow、ONNX 或自定义模型格式，也能够新增评估指标；可维护性要求接口、数据库、适配器和前端组件边界清楚，避免模块间强耦合。

[86] [Heading 2] 2.2 系统总体架构设计

[87] [Normal] 系统采用前后端分离架构。前端通过 Vue3 构建单页应用，使用 Element Plus 提供表单、表格、步骤条、对话框等基础组件，使用 ECharts 完成模型性能图表展示。后端采用 Python Web 服务提供 REST API，围绕模型管理、数据集管理、评估任务和报告管理组织业务接口。评估任务不直接阻塞 Web 请求，而是进入任务队列，由后台执行器完成模型加载、数据批处理、推理计算和结果落库。数据库使用 MySQL 保存用户、模型、数据集、任务、指标结果和报告元数据，大文件则保存在文件目录或对象存储路径中。

[89] [Normal] 图2-1 系统总体架构图

[90] [Normal] 架构设计的核心思想是将“交互流程”和“评估计算”分离，将“模型框架差异”和“统一指标计算”分离。前端只关心用户操作和结果展示，后端接口负责参数校验与状态维护，模型适配层负责不同框架模型的加载差异，评估核心层则面向统一的预测输出计算指标。通过这种分层设计，系统既能保证毕业设计阶段的可实现性，又为后续扩展多框架模型、更多指标和更多部署方式保留接口。

[91] [Heading 2] 2.3 业务流程与数据库设计

[92] [Normal] 评估流程从模型上传开始。用户先上传模型文件并填写模型名称、框架、输入尺寸、类别标签等元数据；随后上传或选择已有数据集，系统对数据格式、标签映射和样本数量进行校验；在参数配置阶段，用户选择评估指标、批大小、运行设备、并行任务数和报告格式；任务启动后，系统将任务写入队列并返回任务编号，后台执行器按照状态流转完成推理、统计和结果保存；最后前端通过轮询或消息通知获取结果，展示图表并支持报告导出。

[100] [Heading 2] 2.4 评价指标体系设计

[101] [Normal] 评价指标体系是系统能够形成客观结论的基础。本文将指标划分为准确性指标、类别级指标、效率指标、资源指标和稳定性指标五类。准确性指标用于衡量总体预测正确情况，类别级指标用于分析不同类别之间的差异，效率指标用于衡量模型推理成本，资源指标用于反映模型运行时对硬件环境的依赖，稳定性指标用于观察模型在扰动、噪声和数据分布变化下的表现。通过分层指标体系，系统可以避免只用单一准确率评价模型所带来的片面性。

[102] [Normal] 对于分类任务而言，混淆矩阵是连接基础指标与错误分析的重要数据结构。系统在评估结束后首先统计真实标签与预测标签的二维矩阵，然后在矩阵基础上计算 Accuracy、Precision、Recall 和 F1-score。这样既能输出总体数值，也能保留类别级错误分布。对于样本类别不平衡的任务，系统同时计算宏平均和加权平均：宏平均对每一类赋予相同权重，能够突出少数类识别能力；加权平均按照类别样本数加权，更接近整体任务表现。

[103] [Normal] 效率和资源指标主要服务于部署决策。一个模型即使准确率较高，如果平均推理时延过长、吞吐量不足或显存占用过高，也可能无法部署到实时业务或边缘设备中。系统在任务执行过程中记录开始时间、结束时间、样本数量和批处理耗时，并结合运行设备信息计算平均时延和吞吐量。资源占用指标在毕业设计实现中以显存和内存峰值为主，后续可以扩展到 CPU 利用率、GPU 利用率和磁盘 I/O。

[104] [Normal] 稳定性指标用于描述模型在复杂场景中的表现。对于图像分类任务，可以通过加入轻微噪声、亮度变化、裁剪或压缩扰动观察模型结果变化；对于文本分类任务，可以通过同义词替换、错别字扰动或句式变化观察模型鲁棒性。本文系统在设计上将扰动评估作为可扩展指标，当前论文重点完成指标接口与展示设计，为后续真实扰动实验预留实现位置。

[109] [Heading 2] 3.1 前端交互与原型设计

[110] [Normal] 前端设计遵循“上传—配置—执行—查看—导出”的主流程。系统左侧为功能导航区，顶部显示系统名称和当前用户信息，右侧为业务操作区域。页面组件按照模型管理、数据集管理、评估任务和结果报告进行拆分，每个页面只承载一个主要操作目标，避免用户在同一界面中面对过多参数。中期汇报中提出当前界面操作流程较繁琐的问题，因此本文在原型设计中加入步骤引导、参数摘要、状态提示和下一步按钮，使用户能够明确当前所处流程。

[111] [Normal] 模型与数据集上传页面主要解决文件导入和元数据录入问题。模型上传区支持拖拽上传，系统读取模型框架、输入尺寸、版本号和标签数量等信息；数据集配置区提供数据类型、样本划分、标签映射和预处理方式等表单项。对于标签不一致、文件格式错误或样本数量不足等异常，前端应在提交前完成基础提示，后端再进行严格校验。

[123] [Heading 2] 3.2 模型与数据集管理模块

[124] [Normal] 模型管理模块的主要任务是将不同来源的模型文件转化为系统可识别、可调度、可追踪的模型资源。用户上传模型后，系统首先校验文件大小和扩展名，然后根据用户选择或文件特征确定模型框架。对于 PyTorch 模型，系统记录权重文件路径、模型类名称、输入尺寸和标签集合；对于 ONNX 模型，系统记录输入节点、输出节点和推理后端；对于未来可能支持的 TensorFlow 模型，系统预留适配器接口。模型记录入库后，系统生成唯一版本号，便于同一模型不同训练轮次之间进行对比。

[125] [Normal] 数据集管理模块负责接收图像、文本或表格形式的数据，并完成样本校验、标签映射和基础预处理配置。以图像分类为例，数据集可以采用按类别目录组织的形式，也可以采用图片路径与标签表组合的形式。系统需要检查标签是否为空、类别数量是否与模型输出一致、图片是否可读取、样本数量是否满足评估要求。对于文本分类数据集，系统需要检查字段名、编码格式和空值情况。通过统一的数据集描述结构，评估核心可以以批处理方式读取样本并返回真实标签与预测标签。

[126] [Normal] 为降低模块耦合，系统将模型加载逻辑封装到 ModelAdapter，将数据读取逻辑封装到 DatasetLoader。评估任务只依赖统一接口，不直接关心底层模型框架和数据格式。当系统新增框架或数据类型时，只需要新增适配器或加载器，并在配置表中注册即可。这一设计回应了中期汇报中“多框架模型兼容性不足”的问题。

[127] [Heading 2] 3.3 评估核心模块设计

[128] [Normal] 评估核心模块是系统的关键部分，主要包括任务调度、模型推理、指标计算和结果持久化四个环节。用户启动评估后，Web 服务生成任务记录并将任务状态置为 waiting；后台执行器获取任务后置为 running，加载模型和数据集并按照批大小进行推理；每个批次产生预测结果后，系统累计真实标签、预测标签和推理耗时；全部样本完成后，指标计算器统一计算 Accuracy、Precision、Recall、F1-score、混淆矩阵、平均时延、吞吐量和资源消耗。

[129] [Normal] 在指标计算方面，Accuracy 反映总体分类正确比例，Precision 反映被预测为某类别的样本中真正属于该类别的比例，Recall 反映某类别真实样本被正确识别的比例，F1-score 是 Precision 与 Recall 的调和平均。对于类别不平衡数据集，系统提供宏平均和加权平均两种方式。混淆矩阵用于定位具体类别之间的误判关系，能够辅助研发人员发现标签相似、数据质量不足或模型表达能力有限等问题。

[130] [Normal] 任务异常处理是评估核心模块稳定性的重点。系统需要捕获模型加载失败、数据读取失败、标签不匹配、显存不足、任务超时和指标计算异常等情况，并将错误信息写入任务日志。前端展示时不直接暴露复杂堆栈，而是给出可理解的处理建议，例如“请检查模型输出类别数是否与数据集标签数一致”。这样的异常设计可以提升系统可用性，也便于后续定位问题。

[136] [Normal] 安全性方面，系统需要限制上传文件类型和大小，避免用户上传无关文件或恶意脚本；需要对接口进行鉴权，避免未授权用户访问他人模型与报告；需要对任务日志进行脱敏处理，避免在报告中暴露本地路径、账号和密钥信息。虽然毕业设计系统规模有限，但在设计阶段加入这些约束，有助于提升系统的工程完整性。

[137] [Heading 2] 3.5 关键流程实现说明

[138] [Normal] 评估任务创建流程是系统业务闭环的起点。前端提交模型编号、数据集编号和评估配置后，后端首先验证模型和数据集是否存在，再检查模型输出类别数是否与数据集标签映射一致。如果检查通过，后端创建评估任务记录，写入 config_json 字段保存批大小、指标列表、设备类型和报告格式等参数，然后将任务编号发送到后台队列。接口立即返回任务编号，前端进入任务进度页面，避免长时间等待造成页面阻塞。

[139] [Normal] 后台执行器采用状态机思想管理任务生命周期。任务初始状态为 waiting，被执行器获取后变为 running；如果模型加载、数据读取、推理计算和指标统计均成功，则状态变为 success；如果任一环节出现异常，则状态变为 failed，并写入错误摘要和详细日志。状态流转清晰后，前端只需要根据状态展示不同组件：waiting 显示排队提示，running 显示进度条，success 展示结果图表，failed 展示错误原因和重新配置入口。

[140] [Normal] 指标计算流程保持独立性和可测试性。系统将真实标签 y_true、预测标签 y_pred、预测概率 y_prob 和耗时信息作为指标计算器的输入，不直接依赖模型对象。这样做的好处是模型推理与指标计算可以分别测试：模型适配器只保证输出格式正确，指标计算器只关注数学统计逻辑。对于新增指标，只需要在指标注册表中声明指标名称、计算函数和展示类型，前端即可根据展示类型选择卡片、表格、柱状图或矩阵图。

[141] [Normal] 报告生成流程强调可追溯性。报告中不仅包含最终指标，还记录模型名称、模型版本、数据集名称、样本数量、标签映射、评估参数、运行时间和异常说明。用户再次查看历史任务时，可以确认当时使用的模型和数据集版本，避免“同名模型覆盖”“数据集被修改”导致报告无法复现。对于毕业设计答辩场景，这种报告也能够清晰展示系统的完整业务链路。

[142] [Normal] 日志与异常处理贯穿整个实现过程。模型文件不存在、文件格式不支持、标签映射不一致、显存不足、任务超时、报告生成失败等情况都需要被捕获并转换为可理解提示。系统设计中将错误分为用户可修复错误和系统运行错误两类：前者提示用户修改文件或参数，后者提示管理员查看日志。通过这种分类，系统不会把底层异常直接暴露给普通用户，也便于维护人员快速定位问题。

[166] [Heading 2] 4.4 性能与兼容性分析

[167] [Normal] 性能测试表明，系统评估耗时主要由模型推理和数据读取两部分组成。当批大小较小时，任务调度和 I/O 开销占比较高；当批大小增大后，GPU 计算效率提升，但显存占用也随之增加。因此系统在参数配置页面提供批大小和并行任务数调节，并在任务摘要中提示预计耗时和资源需求。对于大规模数据集，系统应采用分批读取和结果增量写入，避免一次性加载全部样本导致内存压力过大。

[178] [Heading 2] 5.1 系统优化方向

[179] [Normal] 中期汇报中提出了三个需要继续优化的问题：界面操作流程较为繁琐、多框架模型兼容性不足、核心功能单元测试和稳定性仍需加强。针对界面问题，本文在原型中采用步骤条、任务摘要、默认参数模板和异常提示优化用户路径，使用户在上传模型后能够顺畅进入数据集配置和评估启动页面。针对兼容性问题，本文设计模型适配器接口，将不同框架模型的加载、输入转换和输出解析封装在独立类中，评估核心只处理统一预测结果。针对稳定性问题，系统需要进一步补充单元测试、接口测试和任务异常测试，覆盖正常流程与边界场景。

[180] [Normal] 后续还可以从缓存机制、分布式任务执行和指标插件化三个方向提升系统能力。缓存机制可以保存已完成任务的中间预测结果，避免用户仅修改展示方式时重复推理；分布式任务执行可以将多个评估任务分配到不同计算节点，提高大规模评估效率；指标插件化可以允许用户按照统一接口编写自定义指标，使系统适应更多业务需求。对于资源消耗评估，还可以接入 GPU 利用率、显存峰值、CPU 占用、磁盘 I/O 等监控数据，使系统输出更接近真实部署表现。

[188] [Normal] 本文围绕智能分类模型性能评估中指标单一、流程分散、结果不易比较和可视化不足等问题，设计并实现了一套面向智能分类模型的性能评估系统。系统在需求分析基础上划分为模型管理、数据集管理、评估核心、结果可视化和系统管理五大模块，采用前后端分离架构组织业务流程，并通过模型适配器、任务队列和指标计算器实现评估流程的自动化与可扩展。

[189] [Normal] 在系统设计方面，本文给出了总体架构、业务流程、数据库 ER 关系和核心表结构；在实现方面，重点说明了 Vue3 前端原型、模型与数据集管理、评估核心、可视化报告和 Docker 化部署思路；在测试方面，设计了功能测试、性能测试和多模型评估实验，并通过指标表、柱状图、混淆矩阵和雷达图展示系统输出效果。实验分析表明，系统能够较好支持模型上传、数据集校验、参数配置、任务执行、结果展示和报告导出，能够为分类模型选型和优化提供较直观的依据。

[190] [Normal] 受毕业设计时间和实验条件限制，本文系统仍有进一步完善空间。例如，目前多框架适配主要围绕 PyTorch 和 ONNX 展开，对 TensorFlow、联邦学习模型和增量学习模型支持不足；资源消耗评估仍以基础指标为主，尚未形成完整的部署监控闭环；系统测试数据规模有限，后续需要在更多公开数据集和真实业务数据上验证。未来工作可以继续完善指标插件机制、分布式任务执行、报告模板定制和模型解释能力，使系统更好地服务于真实智能应用场景。
